# 雾霾探测系统设计实验报告

## 一 问题描述

雾霾和空气污染会影响人们的出行安排与身体健康。用户在出门前需要及时了解所在城市的天气状况、空气质量指数、主要污染物浓度以及未来天气变化，从而判断是否需要佩戴口罩、减少户外运动或调整出行计划。

本实验设计并实现一套手机端雾霾探测系统。系统需要完成以下目标：

1. 通过手机定位获取用户所在城市，并将定位城市保存到服务器端，同时显示在手机客户端。
2. 通过天气界面动态显示当前天气、空气质量指数和主要污染物数据。
3. 定位城市由服务器端保存后，天气详情界面通过该城市或行政区划代码请求百度天气 API，获取天气详情和空气质量指数，并在服务器端保存最近一次天气结果。
4. 手机端展示温度、湿度和 AQI 趋势折线图，并标明各条线含义。
5. 在原有天气和空气质量功能基础上，进一步展示未来 7 天预报、未来 24 小时逐小时预报和生活指数，使界面更接近手机自带天气应用。

本系统的重点是将定位、服务器端存储、第三方天气数据获取和手机端可视化展示整合成一个完整闭环。

## 二 方案设计

系统采用“Android 手机客户端 + Spring Boot 服务端”的前后端分离方案。

Android 客户端使用原生 Java 开发，负责申请定位权限、获取手机系统定位、提交定位数据、请求天气数据和渲染界面。客户端界面参考手机系统天气应用的常见结构，将当前城市、当前温度和天气摘要放在顶部，下方依次显示逐小时预报、未来 7 天预报、空气质量、污染物监测、生活指数和趋势折线图。

Spring Boot 服务端使用 Maven 管理，负责提供 REST API、保存客户端定位城市、维护客户端 session、调用百度逆地理编码接口和百度天气接口，并将天气数据整理成统一 JSON 响应返回给客户端。

系统主要模块如下：

| 模块 | 位置 | 作用 |
| --- | --- | --- |
| 定位模块 | Android `MainActivity` | 调用 `LocationManager` 获取 GPS 或网络定位 |
| 网络请求模块 | Android `WeatherClient` | 请求 `/api/location` 和 `/api/weather` |
| 趋势图模块 | Android `TrendChartView` | 绘制温度、湿度、AQI 折线图和颜色图例 |
| 定位服务 | Spring Boot `LocationService` | 保存定位城市、区县、坐标和行政区划代码 |
| 天气服务 | Spring Boot `WeatherService` | 组织天气、AQI、预报、生活指数数据 |
| 百度天气服务 | Spring Boot `BaiduWeatherService` | 调用百度天气 API 并解析返回结果 |
| 百度逆地理编码服务 | Spring Boot `BaiduGeocodingService` | 根据经纬度解析城市、区县和 adcode |
| 客户端会话服务 | Spring Boot `ClientSessionService` | 为多台手机保存独立定位状态 |

系统数据流程如下：

1. 用户打开 App，客户端读取本地保存的城市和服务器地址。
2. 用户点击“手机定位”，客户端申请定位权限并读取经纬度。
3. 客户端调用 `POST /api/location`，把城市名和经纬度提交给服务端。
4. 服务端调用百度逆地理编码接口，解析城市、区县、展示名称和行政区划代码。
5. 服务端按客户端 session 将定位信息保存到 `data/store-{session}.properties`。
6. 客户端调用 `GET /api/weather` 获取天气数据。
7. 服务端优先使用已保存的定位城市和行政区划代码调用百度天气 API。
8. 服务端返回当前天气、AQI、污染物、未来 7 天预报、未来 24 小时逐小时预报、生活指数和折线图数据。
9. 客户端将数据渲染到对应卡片和趋势图中。

为了支持多台手机同时使用同一个服务端，服务端设计了客户端 session 机制。第一次请求时，服务端会生成随机 UUID，并通过 `X-Client-Session-Id` 响应头和 `Haze-Session` Cookie 返回给客户端。Android 端将 session 保存到 `SharedPreferences`，后续请求继续携带该 session。这样不同手机的定位城市不会互相覆盖。

## 三 数据获取

### 1. 定位数据获取

客户端通过 Android 系统 `LocationManager` 获取定位。系统优先读取 GPS 或网络定位的最近位置，如果没有可用位置，则请求一次定位更新。定位结果包含经纬度，客户端将其封装成 JSON 提交给服务端：

```json
{
  "city": "北京",
  "coords": {
    "lat": 39.9042,
    "lon": 116.4074
  }
}
```

服务端收到定位坐标后调用百度地图逆地理编码接口，解析 `city`、`district`、`displayName` 和 `adcode`。其中 `adcode` 会作为百度天气接口的 `district_id` 参数，用于查询对应区域天气。

### 2. 天气和空气质量数据获取

服务端通过百度天气查询接口获取天气和空气质量数据，请求格式如下：

```text
https://api.map.baidu.com/weather/v1/?district_id=行政区划代码&data_type=all&ak=百度AK&sn=签名
```

本项目支持百度 AK/SK 配置。若 AK 开启了 `sn` 校验，服务端会根据 SK 自动生成签名。天气 API 返回后，服务端解析以下数据：

| 数据类型 | 百度字段 | 项目展示 |
| --- | --- | --- |
| 当前天气 | `result.now.text` | 天气现象 |
| 当前温度 | `result.now.temp` | 当前温度 |
| 体感温度 | `result.now.feels_like` | 天气详情 |
| 湿度 | `result.now.rh` | 天气详情、折线图 |
| 风力风向 | `wind_class`、`wind_dir` | 天气详情、逐小时预报 |
| 空气质量 | `aqi` | AQI 卡片 |
| 污染物 | `pm25`、`pm10`、`co`、`no2`、`so2`、`o3` | 污染物监测卡片 |
| 未来 7 天 | `result.forecasts` | 多日预报 |
| 未来 24 小时 | `result.forecast_hours` | 逐小时预报 |
| 生活指数 | `result.indexes` | 生活指数卡片 |

### 3. 服务器端保存

定位数据保存到服务端 `data/store-{session}.properties`，主要包括：

```text
city=城市
district=区县
displayName=城市 · 区县
lat=纬度
lon=经度
districtId=行政区划代码
updatedAt=更新时间
```

天气结果会保存最近一次记录到 `data/last-weather.txt`。这样可以证明天气和空气质量数据经过服务端获取和处理，不是客户端直接请求第三方接口。

### 4. 兜底数据

考虑到实验环境中可能出现网络异常、百度 AK 未配置、百度接口失败或行政区划代码缺失等情况，服务端设计了兜底模拟数据。模拟数据会生成当前天气、AQI、污染物、未来 7 天预报、未来 24 小时预报和生活指数，保证客户端界面不会空白，便于实验演示和功能测试。

## 四 结果展示及分析

系统运行后，手机端界面由多个清晰的信息区域组成，整体结构接近手机自带天气应用。

### 1. 当前天气概览

首页顶部使用深色渐变背景显示当前城市、当前温度、天气现象、体感温度和 AQI。用户点击“手机定位”后，系统会读取手机定位并同步到服务端；点击“刷新数据”后，系统会重新请求天气数据。

该部分实现了天气应用中最核心的信息展示，使用户进入页面后可以第一时间看到当前位置和当前天气状态。

### 2. 逐小时预报

逐小时预报采用横向滑动卡片展示未来 24 小时天气，每个卡片包含：

- 时间
- 天气符号
- 温度
- 天气文字
- 湿度

该区域可以帮助用户了解当天后续时间段的温度、湿度和天气变化。例如用户可以根据后续是否降温、是否下雨来决定是否携带外套或雨具。

### 3. 未来 7 天预报

未来 7 天预报以列表形式展示日期、星期、白天天气、夜间天气、风向风力、最低温和最高温。该部分来自百度 `forecasts` 字段，能够展示更长时间范围的天气趋势。

通过多日预报，用户可以判断未来几天是否有降温、降雨、阴天或晴天等变化，为出行和生活安排提供参考。

### 4. 空气质量指数

空气质量指数模块显示 AQI 数值、空气质量等级和出行建议。系统根据 AQI 值进行分级，常见分级如下：

| AQI 范围 | 等级 | 建议 |
| --- | --- | --- |
| 0-50 | 优 | 适合户外活动 |
| 51-100 | 良 | 敏感人群减少长时间户外运动 |
| 101-150 | 轻度污染 | 敏感人群建议佩戴口罩 |
| 151-200 | 中度污染 | 儿童与老人减少外出 |
| 201-300 | 重度污染 | 尽量留在室内 |
| 300 以上 | 严重污染 | 避免外出 |

该模块是雾霾探测系统的核心功能之一，能够直接反映当前空气质量情况。

### 5. 污染物监测

污染物监测模块保留并展示以下指标：

- PM2.5
- PM10
- CO
- NO₂
- SO₂
- O₃

相比只显示 AQI，污染物详情可以更全面地说明空气污染来源和程度。例如 PM2.5 和 PM10 与雾霾关系较强，O₃ 则常见于光化学污染场景。

### 6. 生活指数

生活指数模块以双列卡片展示百度返回的 `indexes` 数据，包括：

- 穿衣指数
- 运动指数
- 紫外线指数
- 洗车指数
- 感冒指数
- 晨练指数

每个生活指数包含名称、简要等级和详细建议。该模块使系统从单纯的数据展示扩展到生活建议展示，提高了应用实用性。

### 7. 温度、湿度和 AQI 趋势图

趋势图使用 Android 自定义 View 绘制折线图，展示温度、湿度和 AQI 的变化情况。图例使用不同颜色标明曲线含义：

- 橙色：温度
- 青绿色：湿度
- 红紫色：AQI

折线图让用户能够更直观地观察天气和空气质量变化趋势，比单个数值更容易理解。

### 8. 结果分析

从功能完成度看，系统已经满足实验要求：

1. 定位城市能够保存到服务端，并在客户端显示。
2. 手机端界面包含天气、空气质量、污染物、生活指数、逐小时预报、多日预报和趋势图，展示内容完整。
3. 服务端能够使用定位城市和行政区划代码调用百度天气 API 获取天气详情和 AQI。
4. 服务端保存定位数据和最近一次天气结果，符合“服务器端保存”的要求。
5. 当百度接口不可用时，系统提供兜底模拟数据，保证实验展示稳定。

从界面效果看，系统把主要天气信息放在顶部，把逐小时和多日预报放在中间，把服务器设置和手动查询放在页面底部，避免调试入口影响天气信息展示，整体更接近手机自带天气应用。

## 五 心得与体会

通过本次实验，我对移动端应用、后端服务和第三方 API 协同工作的流程有了更完整的理解。手机端不仅要负责界面展示，还要处理定位权限、网络请求、数据解析、异常兜底和屏幕适配；服务端则需要完成数据持久化、第三方接口调用、请求签名、session 管理和响应封装。

在实现百度天气接口调用时，我认识到真实项目中第三方 API 往往存在权限、签名、字段差异和网络异常等问题。因此系统不能只考虑理想返回结果，还需要设计合理的异常处理和模拟数据兜底机制。

在界面设计方面，天气应用需要同时展示大量信息，如果排版不合理会影响阅读效率。本系统通过当前天气概览、逐小时预报、7 天预报、AQI、污染物、生活指数和趋势图分层展示，使用户可以先看到最重要的信息，再按需查看详细数据。

本实验提升了我对 Android 原生开发、Spring Boot 服务端开发、REST API 设计、JSON 数据解析、百度地图开放平台接口调用和天气数据可视化的综合实践能力。后续如果继续完善，可以加入历史空气质量记录、真实未来 AQI 预报接口、地图选点、更多图表交互以及后台定时刷新功能。

## 附录：运行与测试

### 服务端运行

```bash
cd maven-server
mvn spring-boot:run
```

如需使用百度真实数据，可配置：

```bash
BAIDU_MAP_AK=你的百度AK
BAIDU_MAP_SK=你的百度SK
```

项目也支持在根目录或 `maven-server/` 目录下使用 `.env` 文件配置 AK/SK。

### Android 编译

在 Windows 或 Android Studio 中进入 `android/` 目录执行：

```bat
gradlew.bat :app:assembleDebug
```

### 主要接口

| 接口 | 方法 | 作用 |
| --- | --- | --- |
| `/api/location` | GET | 获取当前客户端保存的定位 |
| `/api/location` | POST | 保存手机定位坐标和城市 |
| `/api/weather` | GET | 获取天气、AQI、污染物、预报和生活指数 |
| `/api/reverse-geocode` | GET | 根据经纬度解析城市和区县 |
